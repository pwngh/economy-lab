## Why a signing seam

A hash chain proves the ledger is _internally consistent_: that nobody edited a posting without the head hashes giving it away. It can't prove the chain you're looking at is the one the platform actually wrote. Someone with write access could rewrite a whole account's history, recompute every head, and present a chain that re-derives cleanly.

A signature closes that gap. The platform periodically signs a snapshot of the ledger with a secret only it holds. An auditor with the matching public key can then confirm two things at once: the snapshot is intact, and it came from the platform. A forged chain won't carry a valid signature.

The <ApiLink to="interfaces/src_ports.Signer" /> port is that seam. The core never holds a key or picks an algorithm. It asks the port to `sign` some bytes and, later, to `verify` them. This is what lets the [signed checkpoint](/economy/concepts/integrity/) be tamper-evident rather than merely self-consistent.

## The interface

The port is two required methods and one optional one, all working on raw bytes (see <SourceLink to="src/ports.ts#L81-L93">source</SourceLink>):

```ts
interface Signer {
  sign(bytes: Uint8Array): Promise<Uint8Array>;
  verify(bytes: Uint8Array, signature: Uint8Array): Promise<boolean>;
  kid?(): Promise<string>;
}
```

`sign` always signs with the current key. `verify` returns `true` when the signature matches the current key or any still-valid older one. Because older keys still verify, a signature made before a rotation keeps verifying afterward. `kid` names the key `sign` currently uses; the default signer answers the first 16 hex characters of its Ed25519 public key. The seal stamps it onto each checkpoint row.

<Callout type="note">
  The `kid` is audit metadata, never a verification input, and no key material
  crosses the port. `verify` simply tries every key it trusts until one
  matches, so tampering with a stored `kid` changes nothing about whether the
  signature verifies. A signer without `kid` seals rows with a null one; key
  selection lives entirely inside the adapter.
</Callout>

## The default signer

The reference adapter, <SourceLink to="src/runtime.ts#L134-L171">systemSigner</SourceLink>, signs with [Ed25519](https://ed25519.cr.yp.to/): the configured secret derives a private key that produces 64-byte signatures.

You construct it with a current signing key and an optional list of prior keys:

```ts
const signer = systemSigner({
  signingKey: hexSecret,
  priorKeys: [previousHexSecret],
});
```

The secret is hex-encoded key bytes the host loads from its own config and passes in. This module never reads it from a global or the environment. A misconfigured deploy fails at startup, not deep inside a request. In a wired economy the secret arrives as `signingSecret`; see the [configuration reference](/economy/reference/configuration/#required-secrets) for where it's loaded.

To publish the verification key, `signingPublicKeyHex` derives the hex-encoded Ed25519 public key from the same secret. An external party uses it to verify signed checkpoints without ever seeing the signing secret.

<Callout type="warning">
  The webhook HMAC seam is a separate thing. Inbound provider callbacks are
  authenticated with an HMAC-SHA256 over the raw body using `webhookSecret`.
  That's request authentication at the HTTP boundary, not the `Signer` port. The
  signer signs the ledger's own checkpoints with `signingSecret`.
</Callout>

## How key rotation works

The interface has no rotation call. You rotate by moving the current secret into `priorKeys` and setting a fresh `signingKey`:

```ts
const signer = systemSigner({
  signingKey: newHexSecret,
  priorKeys: [oldHexSecret],
});
```

From that point, every new checkpoint is signed with the new key. Old checkpoints stay verifiable because the old key is still in `priorKeys`. `verify` walks its trusted public keys in order and returns `true` on the first that matches, so a signature from any retained key still passes.

Drop a key from `priorKeys` only once no checkpoint you still want to verify was signed under it; its signatures then stop verifying.

Each checkpoint records the `kid` of the key that sealed it, so after a rotation an operator can tell from the rows alone which checkpoints predate the new key — no signature-by-signature testing needed.

## What the checkpoint relies on it for

The signer has exactly one caller in the core: the checkpoint. The per-account hash chain doesn't use it. Chain links are _hashed_, not signed, so the signer never touches a [posting](/economy/concepts/accounts-and-double-entry/) or an account head directly.

Instead, [the background worker](/economy/reference/background-worker/) periodically seals a checkpoint. <SourceLink to="src/chain.ts#L288-L331">recordCheckpoint</SourceLink> collects every account's head hash and reduces them to a single Merkle root. It then signs _that one root_, so one signature covers every account at once.

The proof comes before the signature: `recordCheckpoint` first runs [the chain prover](/economy/concepts/the-proof/) over every account. If a chain fails to re-derive, it throws `CHAIN_BROKEN` and persists nothing.

Verification runs the same machinery in reverse. <SourceLink to="src/chain.ts#L347-L382">verifyCheckpoint</SourceLink> recomputes the Merkle root over the current heads and compares it to the checkpoint's stored root. When those match, it calls `signer.verify` over that root. A mismatch (a changed root, a dropped account, or an inauthentic signature) returns `false`.

## Out of scope

The port deliberately doesn't cover a few things:

- **Key storage and loading.** The adapter receives ready secrets; where they come from (a secrets manager, an env var) is the host's job.
- **A rotation schedule.** Rotation is a redeploy with new key arguments; nothing in the port drives it on a timer.
- **Invalidating a past signature.** The `kid` on each checkpoint records which key sealed it, but that is audit metadata only — retiring a key from `priorKeys` is the only way its signatures stop verifying.
- **Choosing where roots are anchored.** The [`Anchor` port](/economy/reference/offline-verification/) pushes each sealed checkpoint to an external store, but which store — and running it — is the operator's job.
