An `Economy` is the object you hold and call. It has a small surface: one `submit` to change money, a `read` group to look at state, and `close` to shut down.

You build it from a set of injected ports (a `Store`, a `Signer`, a `Processor`, a `Rates`, a `FeePolicy`, a clock, and an id source) so the core never reaches for a database driver or a wall clock. Pass in-memory ports to run the same economy with no external infrastructure.

## Construction

`createEconomy(options?)` is the one call. With nothing, it builds a batteries-included in-memory economy: the in-memory `Store`, a dev `Signer`, a dev `Rates` table, a flat `FeePolicy`, and an in-memory `Processor`, all wired. No environment, no infrastructure:

```ts
import { createEconomy } from '@pwngh/economy-lab';

const economy = await createEconomy();
```

Pass `env` and it resolves everything from the environment instead: the `Store` from `DATABASE_URL` (a `postgres://` or `mysql://` connection string picks that engine, and an unset value falls back to the in-memory store), and the four external ports and config from their own keys. Outside production it fills sane defaults for anything absent; in production it requires the real values and fails fast with one message naming what is missing. Any field you pass explicitly wins over the env-derived one:

```ts
const economy = await createEconomy({ env: process.env });

// env for everything, but override one port:
const withLiveRates = await createEconomy({ env: process.env, rates: liveRates() });
```

When a host already holds a full `Capabilities` bundle вЂ” for instance to share one `Store` between an economy and a background worker вЂ” it calls the low-level assembler directly:

```ts
const economy = economyFromCapabilities(capabilities);
```

Each driver is imported only when its environment variable selects it, so a deployment installs only what it uses. See [configuration](/economy/reference/configuration/) for every variable `compose` reads.

## submit

`submit(operation)` applies one [`Operation`](/economy/reference/operations/) and resolves to an [`Outcome`](/economy/reference/outcomes-and-reason-codes/). The operation names the action and carries an `idempotencyKey`, so a retried request runs at most once.

<SubmitPipeline />

Here a user tops up their balance; the result tells you whether it `committed`, repeated an earlier request (`duplicate`), or was declined (`rejected`):

```ts
const outcome = await economy.submit(
  topUp({
    idempotencyKey: 'ord_8821',
    actor: systemActor('billing'),
    userId: 'usr_a1',
    amount: toAmount('CREDIT', 5000n),
    source: 'stripe',
  }),
);
// в†’ { status: "committed", transaction: { id: "txn_вЂ¦", вЂ¦ } }
```

A `rejected` outcome is a normal "no" returned as data, not an error; a genuine fault throws instead. See [outcomes and reason codes](/economy/reference/outcomes-and-reason-codes/) for the full set.

Every submit is also metered through the injected `Meter`: one count under `economy.submit`, tagged by operation kind and how it resolved (`committed`, `duplicate`, `rejected`, or `fault`), plus its duration under `economy.submit.ms` вЂ” request-path telemetry without wrapping `submit` yourself.

## read

`read` is the look-but-don't-change surface: balances, statements, a posting or saga by id, entitlement checks, the account, payout, and posting journals, the pause status, and the solvency proof.

Reading one account's balance is a single call that returns an `Amount`:

```ts
import { spendable } from '@pwngh/economy-lab';

const balance = await economy.read.balance(spendable('usr_a1'));
// в†’ { currency: "CREDIT", minor: 5000n }
```

See [reads](/economy/reference/reads/) for the whole surface and what each method returns.

## close

`close()` releases the store's resources (a database pool, an open connection) and resolves when shutdown is done.

```ts
await economy.close();
```

## The request path

Every `submit` runs the same path: validate the operation, authorize the caller, then post the money in one transaction that writes both the ledger entries and any outgoing events.

```text
submit(operation) в”Ђв–¶ validate В· authorize В· post в”Ђв–¶ Ledger (append-only В· hash-chained)
                                                      в”‚
                                  read В· balance в—Ђв”Ђв”Ђв”Ђв”Ђв”¤в”Ђв”Ђв–¶ Outbox (same transaction)
```

Validation rejects a malformed operation before any work begins: a blank `idempotencyKey`, an ownerless wallet account, an out-of-range amount. Authorization decides whether this `actor` may run this kind of operation, so a forbidden request stops before money moves.

The post then runs inside one all-or-nothing transaction. It appends the balanced [double-entry](/economy/concepts/accounts-and-double-entry/) legs to the ledger and enqueues the matching event into the outbox in that same transaction, so the two commit together or not at all; a rollback can't leave a stray event in the outbox.

A `read.balance` returns a stored running total, so it's a single O(1) read rather than a re-sum of history. The ledger entries stay the source of truth: the [prover](/economy/concepts/the-proof/) re-derives each balance and reports drift.

The background sweeps that drain the outbox and do the rest of the deferred work run off this path on their own schedule. See the [background worker](/economy/reference/background-worker/).
