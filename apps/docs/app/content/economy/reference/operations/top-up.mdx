`topUp` turns a buyer's cleared cash into spendable credit. It raises a user's
[`spendable`](/economy/concepts/accounts-and-double-entry/) balance and records the USD that paid for
it.

It runs after a card or wallet charge clears, so the caller is the trusted payment service, not the
buyer. You name the user, the credit `amount`, and the funding `source`:

<CallTabs name="top-up" http={topUpHttp}>

```ts
const outcome = await economy.submit({
  kind: 'topUp',
  idempotencyKey: 'idem_0',
  actor: { kind: 'system', service: 'payments' },
  userId: 'usr_buyer',
  amount: toAmount('CREDIT', 5_000n),
  source: 'card',
});
// → { status: "committed", transaction: { id: "txn_…", … } }
```

</CallTabs>

The credit `amount` is what the user receives. What they paid in USD is derived from the
[`buy` rate](/economy/concepts/money-model/), not passed in.

## Parameters

The payload fields, beyond the `kind` tag, are:

| Field            | Type                                                   | Default    | Description                                                                                                                   |
| ---------------- | ------------------------------------------------------ | ---------- | ----------------------------------------------------------------------------------------------------------------------------- |
| `idempotencyKey` | `string`                                               | (required) | A retried submit with the same key runs at most once. See [idempotency](/economy/concepts/idempotency/).                        |
| `actor`          | [`Principal`](/economy/concepts/actors-and-authorization/) | (required) | Who is asking. Must be `system` or `operator`.                                                                                |
| `userId`         | `string`                                               | (required) | The user whose `spendable` balance is credited.                                                                               |
| `amount`         | [`Amount`](/economy/concepts/money-model/)             | (required) | The credit to issue. Must be `CREDIT` and strictly positive.                                                                  |
| `source`         | `string`                                               | (required) | The funding rail (`card`, `steam`, …). Selects the credits' maturity horizon and must be non-empty after trimming whitespace. |

The `source` sets how long the new credits are held before they can be
spent or cashed out, see [maturity](/economy/concepts/credit-maturity/). An unrecognized `source` falls
back to the long default horizon, never a fast one.

## Returns

`topUp` resolves to an [`Outcome`](/economy/reference/outcomes-and-reason-codes/).

- `committed`: the credit was issued. The `transaction` carries the issuance posting (the buyer's
  credits going up).
- `duplicate`: the `idempotencyKey` was already used. The earlier transaction is returned unchanged,
  and nothing new posts.
- `rejected`: a velocity or maintenance-window decline; see [reason codes](#reason-codes).

## Postings

A top-up posts two balanced transactions, because one posting can't mix `CREDIT` and `USD`. The
returned `transaction` is the first.

The first posting issues the credit. It raises the buyer's `spendable` balance and records the same
amount against [`STORED_VALUE`](/economy/concepts/accounts-and-double-entry/), the running count of
all credits in circulation:

| Account               | Side   | Amount            |
| --------------------- | ------ | ----------------- |
| `SYSTEM.STORED_VALUE` | debit  | `amount` (CREDIT) |
| `spendable(userId)`   | credit | `amount` (CREDIT) |

The second posting accounts for the cash the buyer paid. The USD splits two ways. The backing value
(`amount × par`) is held in `TRUST_CASH` to cover the new credits. The buy-vs-par gap (the
[platform spread](/economy/concepts/money-model/)) is recognized as revenue in `REVENUE_USD`:

| Account               | Side   | Amount                               |
| --------------------- | ------ | ------------------------------------ |
| `SYSTEM.TRUST_CASH`   | debit  | backing = `ceil(amount × par)` (USD) |
| `SYSTEM.REVENUE_USD`  | debit  | margin = `gross − backing` (USD)     |
| `SYSTEM.USD_CLEARING` | credit | gross = `ceil(amount × buy)` (USD)   |

The `REVENUE_USD` leg posts only when the margin is positive. An exact-par purchase (no spread) stays
a two-leg cash move.

Both conversions round up — the backing so `TRUST_CASH` always covers the spendable balance at par,
the gross so the margin never goes negative. The cost is at most one minor unit.

<Callout type="note">
  The CREDIT `REVENUE` account is untouched by a top-up, so it keeps meaning
  transaction fees only. The newly issued credits offset against `STORED_VALUE`,
  not `REVENUE`. The external app-store cut and VAT happen at the cash-in rail
  before this ledger sees the purchase, and aren't modeled.
</Callout>

## Authorization

`topUp` is restricted to a privileged [`Principal`](/economy/concepts/actors-and-authorization/): a
trusted `system` service (the verified payment path) or a human `operator`. It mints spendable
credit, so an end user can never issue one: a `user` principal is rejected with `AUTH.UNAUTHORIZED`.

In production, the `system` top-up is driven by a verified [processor](/economy/ports/processor/)
webhook. `toTopUp` builds the operation from a cleared `PurchaseEvent`, keyed off the provider's
`eventId` so a replayed webhook dedupes.

## Reason codes

A `topUp` can return one [reason code](/economy/reference/outcomes-and-reason-codes/) as a
`rejected` outcome:

| Code          | When                                                                                                                                         |
| ------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| `RISK_DENIED` | The user's recent top-up volume crossed the configured velocity limit. A top-up counts against the same per-user velocity window as a spend. |

The pause gate applies only to `user` actors, and a `user` can't call `topUp`, so a top-up is
never declined with `ECONOMY_PAUSED`.

A malformed request throws instead:

| Code                   | When                                        |
| ---------------------- | ------------------------------------------- |
| `OP.MALFORMED`         | A blank `source` or a non-`CREDIT` amount.  |
| `MONEY.INVALID_AMOUNT` | A zero or negative amount.                  |

## Preconditions and invariants

A top-up holds the books straight on both currencies:

- The `amount` must be `CREDIT` and strictly positive; a wrong currency or non-positive amount throws.
- Both postings net to zero within their currency, so the ledger stays
  [balanced](/economy/concepts/accounts-and-double-entry/) and conserved.
- `TRUST_CASH` rises by at least `amount × par`, so the credits are
  [backed](/economy/concepts/solvency/) the moment they're issued: trust cash always covers the
  spendable balance valued at par.
- The new credits carry their `source` and top-up time as a lot, which fixes when they
  [mature](/economy/concepts/credit-maturity/) and become spendable and payable.
