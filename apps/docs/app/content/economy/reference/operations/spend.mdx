`spend` runs a marketplace purchase. It charges the buyer and pays the sellers as one balanced [double-entry posting](/economy/concepts/accounts-and-double-entry/).

The same transaction also records a sale under `orderId` and grants the buyer the `sku`.

A purchase with a single seller taking the whole price looks like this:

```ts
const outcome = await economy.submit({
  kind: 'spend',
  idempotencyKey: 'idem_0',
  actor: { kind: 'user', userId: 'usr_buyer' },
  orderId: 'ord_1',
  buyerId: 'usr_buyer',
  sku: 'wrld_pass',
  price: toAmount('CREDIT', 400n),
  recipients: [{ sellerId: 'usr_seller', shareBps: 10_000 }],
});
// → { status: "committed", transaction: { id: "txn_…", … } }
```

The buyer's `promo` balance is drawn first, then `spendable`. The seller is credited to their `earned` balance, and the platform keeps its fee.

A purchase can split the price across several sellers. Each `Recipient` takes a share in basis points (bps), and the shares must sum to `10_000`:

```ts
recipients: [
  { sellerId: 'usr_seller_a', shareBps: 6_000 },
  { sellerId: 'usr_seller_b', shareBps: 4_000 },
];
```

## Parameters

The payload fields, beyond the `kind` tag, are:

| Field            | Type                                                        | Default    | Description                                                                                                       |
| ---------------- | ----------------------------------------------------------- | ---------- | ----------------------------------------------------------------------------------------------------------------- |
| `idempotencyKey` | `string`                                                    | (required) | A retried submit with the same key runs at most once. See [idempotency](/economy/concepts/idempotency/).          |
| `actor`          | [`Principal`](/economy/concepts/actors-and-authorization/)  | (required) | Who is asking. A `user` actor must be the buyer.                                                                  |
| `orderId`        | `string`                                                    | (required) | The unique key for this purchase. A refund looks the sale up by it.                                               |
| `buyerId`        | `string`                                                    | (required) | The user who pays.                                                                                                |
| `sku`            | `string`                                                    | (required) | The item granted on success.                                                                                      |
| `price`          | [`Amount`](/economy/concepts/money-model/)                  | (required) | The total charge. Must be positive and in `CREDIT`.                                                               |
| `recipients`     | [`Recipient[]`](/economy/ports/pricing/)                    | `[]`       | Sellers and their shares in bps. An empty list means the platform keeps the whole net.                            |
| `ageRestricted`  | `boolean`                                                   | `false`    | Tags the posting for audit. The core does not block on age.                                                       |
| `giftTo`         | `string`                                                    | `buyerId`  | Grants the `sku` to this user instead of the buyer. The buyer still pays.                                         |

Each `Recipient` is `{ sellerId: string; shareBps: number }`. A `shareBps` must be `> 0` and `≤ 10_000`, and the shares must sum to exactly `10_000`.

## Returns

`spend` resolves to an [`Outcome`](/economy/reference/outcomes-and-reason-codes/).

- `committed`: on success, with the posted `Transaction`.
- `duplicate`: a retry under the same `idempotencyKey`.
- `rejected`: a well-formed request that can't proceed, with a `RejectionCode`.

## Postings

`spend` posts one balanced [`Transaction`](/economy/concepts/accounts-and-double-entry/), split into a promo-funded part and a spendable-funded part that each balance on their own.

The `spendable`-funded part debits the buyer and lets the [pricing policy](/economy/ports/pricing/) split the credit across sellers and the house:

| Account              | Side   | Amount                      |
| -------------------- | ------ | --------------------------- |
| `spendable(buyerId)` | debit  | `spendable part`            |
| `earned(sellerId)`   | credit | `seller's share of the net` |
| `SYSTEM.REVENUE`     | credit | `fee + rounding leftover`   |

The `promo`-funded part spends down the buyer's marketing grant and funds the sellers from house revenue, since a promo grant is not money the buyer paid:

| Account              | Side   | Amount                             |
| -------------------- | ------ | ---------------------------------- |
| `promo(buyerId)`     | debit  | `promo part`                       |
| `SYSTEM.PROMO_FLOAT` | credit | `promo part`                       |
| `SYSTEM.REVENUE`     | debit  | `total paid to sellers`            |
| `earned(sellerId)`   | credit | `seller's share of the promo part` |

<Callout type="note">
  Only the `spendable`-funded part carries the platform fee. The `promo`-funded
  part is charged no fee, so a purchase paid entirely from `promo` earns the
  platform nothing.
</Callout>

### How the split is computed

For the `spendable`-funded part, the injected [fee policy](/economy/ports/pricing/) builds the credit side through `splitLegs`. The fee comes off the top, the net is divided among the sale's recipients by share, and `REVENUE` receives the fee plus any rounding leftover, so no minor unit of the price is lost.

```ts
// A 1000-credit sale, 30% fee, one seller taking 100%:
//   credit earned(seller)  700
//   credit REVENUE         300   (fee + leftover)
// Credits store negative, so the two lines sum to -1000, the price.
// The spend handler adds the buyer's matching debit, zeroing the posting.
```

The `promo`-funded part carries no fee, so it skips `splitLegs`: sellers are paid from `REVENUE` and the buyer's `promo` is spent down against `PROMO_FLOAT`.

## Authorization

`spend` is open to an end user: the buyer runs it on their own wallet. A `system` or `operator` [actor](/economy/concepts/actors-and-authorization/) may also call it.

A `user` actor must own every account the operation debits, so a buyer may only spend from their own `promo` and `spendable` balances. Submitting a `spend` whose `buyerId` is someone else's wallet is unauthorized.

## Reason codes

A well-formed `spend` on a healthy system can be declined for these reasons, each returned as a `rejected` [`Outcome`](/economy/reference/outcomes-and-reason-codes/):

| Code                 | When                                                                                                                                  |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| `INSUFFICIENT_FUNDS` | The buyer can't cover the price from `promo` plus `spendable`.                                                                        |
| `FUNDS_IMMATURE`     | The buyer has the credits, but the `spendable` part would dip into funds still in a [settlement hold](/economy/concepts/credit-maturity/). |
| `DUPLICATE_ORDER`    | A sale already exists for this `orderId` under a different `idempotencyKey`, so a second charge is refused.                           |
| `RISK_DENIED`        | The purchase would push the buyer past their recent-spending [velocity limit](/economy/concepts/spend-velocity/).                     |
| `ECONOMY_PAUSED`     | A maintenance window is in effect, so an end user's purchase is declined. The decline carries `resumesAt`.                            |

A malformed request throws instead:

| Code                   | When                                                         |
| ---------------------- | ------------------------------------------------------------ |
| `MONEY.INVALID_AMOUNT` | The `price` is non-positive.                                 |
| `OP.MALFORMED`         | The `price` is not in `CREDIT`.                              |
| `OP.MALFORMED`         | The `sku`, `orderId`, or `giftTo` is blank.                  |
| `OP.MALFORMED`         | The recipient shares don't sum to `10_000`.                  |
| `OP.MALFORMED`         | Two recipients name the same `sellerId`.                     |
| `OP.MALFORMED`         | A recipient names a house account rather than a user wallet. |
| `OP.MALFORMED`         | A recipient equals the `buyerId` (self-dealing).             |

## Preconditions and invariants

The pipeline does the shared work before the handler runs. In order, it:

1. Authorizes the buyer.
2. Drops exact retries by `idempotencyKey`.
3. Screens risk and funds.
4. Locks the affected accounts.

The handler then validates its own fields and posts.

The posted transaction is balanced: every leg's debits and credits sum to zero, both for the `promo`-funded part and the `spendable`-funded part.

The buyer always pays from `promo` first, then `spendable`. The up-front funds check and the posting use the same split, so they can't disagree.

The `sku` entitlement is granted in the same transaction as the charge. It goes to the buyer, or to `giftTo` when that field is present. Because the grant shares the transaction, a rolled-back charge grants nothing.

<Callout type="warning">
  `orderId` is distinct from `idempotencyKey`. The key drops exact retries;
  `orderId` identifies the purchase a
  [refund](/economy/reference/operations/refund/) reverses. Two requests sharing
  an `orderId` under different keys are refused with `DUPLICATE_ORDER`, so the
  buyer is never double-charged for one order.
</Callout>
