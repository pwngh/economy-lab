A <ApiLink to="variables/src.topUp" /> or a <ApiLink to="variables/src.spend" /> finishes the instant you call it. A payout can't: it reaches an external rail that answers in its own time.

So economy-lab models a payout as an explicit state machine. The operation becomes a stored record (a <ApiLink to="interfaces/src_ports.Saga" />) that the [background worker](/economy/reference/background-worker/) advances one safe step at a time.<Cite n={1} /> If a crash interrupts the work, the record resumes from its last recorded state. A step that runs twice still counts once.

## Why it exists

The alternative — paying synchronously inside <ApiLink to="variables/src.requestPayout" /> — couples a money posting to a network call the ledger doesn't control.

If the call hangs, the request hangs. If the process dies after the provider pays but before the ledger records it, the books and the provider disagree, with no record of why.

A long-lived record makes every intermediate state durable and named: the worker can drive a payout forward from wherever it stopped, and an operator sees a stuck one rather than a silent gap.

The same shape covers anything that outlives one request. [Subscriptions](/economy/concepts/subscriptions/) reuse it to renew period after period, and a promo grant's expiry is likewise a row the worker sweeps, not a thread it holds open.

## The states

`requestPayout` does not pay. It reserves the credits (`debit earned → credit payout_reserve`, two [`CREDIT` legs](/economy/concepts/accounts-and-double-entry/) that cancel out) and opens a `Saga`, a record whose `state` walks through `SagaState` (see <SourceLink to="src/ports.ts#L1120-L1126 · SAGA_STATES" />).

The full enum declares `REQUESTED`, `RESERVED`, `SUBMITTED`, `SETTLED`, and `FAILED`. A live payout starts at `RESERVED`, because the credits were set aside in the same database transaction that opened the saga. Watch the row appear:

## The two triggers

The two forward steps run on different triggers. The worker's `payouts` sweep handles only the first. It claims due `RESERVED` sagas, submits the saga's stored USD quote — priced once at request — via `processor.submitPayout`, records the `providerRef`, and advances `RESERVED → SUBMITTED` (see <SourceLink to="src/worker/payouts.ts#L342-L363 · submitToProvider" />).

It does not settle itself. Settlement arrives on the provider's verified "payout settled" webhook. That webhook maps to a [`settlePayout`](/economy/reference/operations/settle-payout/) operation, which the worker's inbox drain applies off the request path. That operation runs the `SUBMITTED → SETTLED` step.

Failure arrives the same way, and promptly. The common real-world failure is asynchronous — the rail accepts a payout and only rejects it during processing — so a verified "payout failed" webhook maps to a [`reversePayout`](/economy/reference/operations/reverse-payout/) carrying `providerReported`, which returns the reserve as soon as the rail gives up instead of after the timeout. And when the processor adapter offers the optional `payoutStatus` probe, the sweep consults it on each look at a `SUBMITTED` saga: a provider-reported `FAILED` or `RETURNED` releases the reserve that run, a reported `SETTLED` blocks the force-fail (a lost settlement webhook must never end in a double-pay), and a reported `PENDING` defers the timeout while the rail is still working.

## What makes it safe to re-drive

Three guards let the worker retry a step without paying twice.

- **The call to the rail is idempotent**, keyed by the saga id (`key: saga.id`), so a retried submit still pays once. The settlement webhook is deduped on the provider's event id, so a redelivered settlement settles once.
- **Each state change is a compare-and-set.**<Cite n={2} /> `sagas.advance(id, from, to, patch)` moves a saga only if it is still in `from`, and returns `false` (changing nothing) otherwise, so two sweeps racing the same saga can't both advance it (see <SourceLink to="src/ports.ts#L699-L705 · SagaStore.advance" />).
- **A stuck payout is given up on.** Past `maxPayoutAttempts` failed attempts, or after `maxPayoutAgeMs` in `SUBMITTED` with no settlement webhook and no provider evidence to the contrary, the sweep force-fails it. It flips the saga to `FAILED` and, in the same transaction, posts the exact reverse of the reservation (`debit payout_reserve → credit earned`), returning the credits to the seller (see <SourceLink to="src/worker/payouts.ts#L114-L181 · deadLetter" />).

That force-fail reversal and the operator's `reversePayout` share one compare-and-set guard on the saga. A reversal posts the compensating entry _only if_ its compare-and-set wins. A lost compare-and-set means a concurrent settle or reverse already accounted for the reserve, so posting again would return the credits a second time. An operator can force the same early reversal with `reversePayout`, but only while the money hasn't already left.

## The invariant

A payout moves through its states **once**, in order, no matter how many times the worker re-drives it or how its triggers race.

Every transition is a compare-and-set against the current state, so a step is applied by exactly one writer. The money posting that accompanies a transition commits in the same database transaction as the state flip, so the books and the record never diverge.

A `RESERVED` payout pays at most once, and its reserve is released exactly once: by settlement, by timeout-fail, or by an operator reversal, whichever wins the compare-and-set.

## How it's enforced

The enforcement lives in three places, and you can verify each one in the source.

- **The store's compare-and-set method.** `SagaStore.advance` is a conditional update that returns whether it changed a row, so the handler can detect and stand down on a lost race (see <SourceLink to="src/ports.ts#L699-L741 · SagaStore" />).
- **One transaction per step.** Each forward step posts its ledger entry and advances its record inside `store.transaction`, so a partial step rolls back whole: a flipped state with no posting, or a posting with no flip, can't persist.
- **Idempotent external calls and dedupe.** `processor.submitPayout` is keyed by the saga id (see <SourceLink to="src/ports.ts#L216-L232 · Processor" />), and inbound provider events are deduped by event id before they apply, so a redelivered webhook or a retried submit resolves to the same money move at most once.

## Operations and ports that rely on it

[`requestPayout`](/economy/reference/operations/request-payout/) opens the saga; <ApiLink to="variables/src.settlePayout" /> and <ApiLink to="variables/src.reversePayout" /> move it. The worker's `payouts` sweep and its inbox drain drive transitions off the request path. The opening transaction's `meta.sagaId` names the saga, so the requester can follow it with `read.saga(sagaId)`.

The saga records persist through the `SagaStore` sub-store of the [`Store`](/economy/ports/storage/) port, and a payout's disbursement leaves through the [`Processor`](/economy/ports/processor/) port. The discipline the saga depends on — a compare-and-set on every transition, one transaction per step — is the same machinery that keeps the [ledger's integrity invariants](/economy/concepts/integrity/) intact under concurrency.
